# 方案：移除举一反三/联想链，改为记忆驱动的自然回复

## 背景

cc-soul 当前有两个"主动补充信息"机制：
- **举一反三**（buildExtraThinkAugment）：600+ 行硬编码正则，匹配关键词后注入"顺便说一下：1. 2. 3."格式指令
- **联想链**（associateSync + getAssociativeRecall）：从记忆图谱找关联记忆，拼进 augment

## 问题

### 举一反三的问题
1. **硬编码不可扩展**：20+ 领域的正则规则，每加一个新领域手动写一段
2. **格式僵化**：强制 LLM 输出"顺便说一下：1. 2. 3."，不自然
3. **不个性化**：所有用户看到同样的提示（"Python 相关：注意 GIL"），不基于用户实际情况
4. **与记忆系统重复**：cc-soul 的 facts 已经提供了个性化上下文，举一反三是多余的

### 联想链的问题
1. **被 activation-field 覆盖**：AAM + 激活场的 6 信号融合已经在做关联召回
2. **额外延迟**：associateSync 遍历记忆图谱，增加响应时间
3. **结果质量不稳定**：关联出的记忆可能跟当前话题不相关

### 100 条测试的证据
关掉举一反三 + 联想链后，kimi 的回复**更好了**：
- 推荐火锅时自动提醒"血压高少盐"（从 facts 读到的）
- 聊创业时自动关联"小雨怀孕、预产期明年三月"（从 facts）
- 30 岁感慨时自动综合列出所有成就（从 facts + 记忆）
- 没有了机械的"顺便说一下：1. 2. 3."格式

## 行业对比

| 产品 | 有没有"举一反三" | 怎么做补充信息 |
|------|----------------|--------------|
| **Mem0** | ❌ 没有 | 只提供记忆，LLM 自己决定回复内容 |
| **Zep** | ❌ 没有 | 提供 Context Blocks（facts + entities），LLM 自己关联 |
| **Letta** | ❌ 没有 | core memory 钉在 prompt，LLM 自己用 |
| **ChatGPT** | ❌ 没有 | User Memory 始终注入，模型自己判断何时提及 |
| **cc-soul** | ✅ 有（600行正则） | 硬编码关键词匹配 → 强制格式输出 |

**没有任何主流记忆产品用正则匹配来控制 LLM 的补充信息。**

## 方案

### 删除什么
1. `buildExtraThinkAugment()` — 600+ 行硬编码正则
2. `associateSync()` 调用 — handler-augments 里的 Layer A
3. `getAssociativeRecall()` 调用 — handler-augments 里的 Layer B
4. SOUL.md 里的"顺便说一下"格式指令
5. handler.ts 里的 tipsSection 注入

### 保留什么
1. **facts 始终注入 system prompt** — 用户的核心信息每次都在（已实现）
2. **activation-field 召回** — 相关记忆通过 recall + crystallize 注入（已实现）
3. **AAM 扩展查询** — 种子 + 学到的关联让召回更准（已实现）

### 新增什么
在 system prompt 里加一句轻量提示（替代 600 行正则）：

```
如果你知道用户的相关背景信息，可以在回复中自然地提及——
不要列清单，不要说"顺便说一下"，像朋友聊天一样自然带出来。
```

这一句话让 LLM 自己判断什么时候该补充、补充什么、怎么补充。

### 为什么这样更好

| 维度 | 举一反三（硬编码） | 记忆驱动（新方案） |
|------|------------------|------------------|
| 覆盖面 | 20 个领域（手动维护） | 无限（LLM 自己判断） |
| 个性化 | ❌ 所有用户看同样的提示 | ✅ 基于用户 facts |
| 回复自然度 | ❌ 机械的"1. 2. 3."格式 | ✅ 像朋友聊天 |
| 维护成本 | 高（每个领域写正则） | 零（LLM 能力自带） |
| 与记忆系统协同 | ❌ 独立的硬编码系统 | ✅ 复用 facts + recall |
| 跟行业对齐 | ❌ 只有 cc-soul 这么做 | ✅ Mem0/Zep/ChatGPT 的方式 |

## 风险

1. **LLM 不补充信息**：如果 facts 注入太少或 LLM 太懒，可能完全不做关联。
   → 缓解：确保 facts 质量高、数量够。100 条测试显示 kimi 在有 facts 时自动关联得很好。

2. **补充信息质量不可控**：LLM 可能补充错误的关联。
   → 缓解：facts 里的数据是验证过的，LLM 基于 facts 关联不太会出错。比硬编码正则（"注意 GIL"对所有人说）更准确。

3. **不同 LLM 表现差异**：弱模型可能不会主动关联。
   → 缓解：system prompt 里的轻量提示引导 LLM 这么做。

## 实施步骤

1. 在 prompt-builder.ts 加一句轻量提示（5 分钟）
2. 删除 buildExtraThinkAugment 的 `return null`（恢复不了，直接删函数）
3. 删除 associateSync 和 getAssociativeRecall 的调用
4. 删除 handler.ts 的 tipsSection
5. 删除 SOUL.md 的"顺便说一下"格式指令（已关闭，改为永久删除）
6. 飞书测试验证

## 时间估计

代码改动：30 分钟
测试验证：1 小时
