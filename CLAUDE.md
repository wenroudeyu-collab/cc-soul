# cc-soul 开发原则（CLI 必读）

## 工作方式

1. **先诊断，后改代码**
   看到问题不要立刻改，先写测试验证假设。
   测试数据出来再决定改什么。

2. **不回退，要修**
   改了跌了 → 找原因 → 修 → 不要回退到旧版本
   除非根本性错误

3. **快速测试优先**
   改代码 → sample 645 题（5-25 分钟）→ 涨了 → 全量 1986 题
   跌了 → 继续改，不浪费 1 小时跑全量

4. **避免捷径**
   - 不从 LOCOMO 测试集提取特征（作弊）
   - 不用 soft-weighting 代替 hard-partition
   - 不用多标签代替单标签
   - 不留"无标签池"

## 架构原则

1. **单标签硬分区**
   Memory.primary_c2: string（不是数组）
   召回用 filter 不用 weight
   池子必须真的变小（C2 池平均 30-50 条）

2. **100% 分类覆盖**
   每条记忆必须有 C1/C2/C3
   种子词 + AAM 动态扩展 + LLM 兜底

3. **分类一致性**
   query 和 memory 用同一个分类函数
   先测一致性（> 80%）再改硬分区

4. **循环配合**
   AAM 学习 → 话题分类更准 → 池更小 → 召回更准
   LLM 教 AAM → AAM 自己分类 → 越用越省 LLM
   蒸馏反哺 AAM → AAM 扩展话题词 → 飞轮效应

5. **分治策略**
   每种查询类型用不同算法
   简单事实 → 小池 BM25
   复杂推理 → iterative recall + coverage rerank
   temporal → 跳过 LLM rewrite，保护时间词

## 历史错误教训（不要重犯）

1. **假分层**：用多标签 + soft-weighting 代替真正的单标签硬分区。池子没变小等于没分。
2. **想从 LOCOMO 提取种子词**：这是作弊。种子库必须通用生成，跟测试集完全隔离。
3. **碰到退步就回退**：应该分析哪些题型涨了哪些跌了，保留涨的逻辑修复跌的原因。
4. **全量测试太频繁**：应该先 sample 验证，确认方向对了再全量。每次全量 2 小时，一天只做几次。
5. **蒸馏和话题分类混淆**：蒸馏 L1/L2/L3 是抽象层级（记忆→主题→画像），话题 C1/C2/C3 是分类层级（生活→吃饭→火锅）。两个维度不同，不要混淆。

## 自我反思原则

- **用户批评时要反思，不要找借口。** 批评意味着方向偏了，停下来想清楚再做。
- **不要跟着惯性走。** 做了 3 小时不代表方向对，随时准备换方向。
- **多思维考虑。** 代码是循环配合的，改一处要想清楚影响其他哪些模块。

## 禁止事项

- 从 LOCOMO 数据集提取种子词（作弊）
- 在未验证分类一致性前改硬分区（蒙眼改代码）
- 跌了就回退（应该找原因修复）
- sample 涨 0.5% 就以为是进步（波动范围 ±3%）
- 连续 5 次只跑 sample 不跑全量（过拟合）
- 一套算法通吃所有题型（分治）
- 用 soft-weighting 代替 hard-partition（假分层）
- 多标签代替单标签（假单标签）
- 不经诊断就开始改代码（先诊断后改）

## 测试规范

- 改动前：跑 sample 记录基线
- 改动后：跑 sample 对比
- 每 3 次 sample 跑一次全量
- 全量必须 > 基线才算进步
- benchmark 模式用 `--online --api --llm --sample`

## 架构目标

- Hit@10: 当前 43% → 目标 80%+
- LLM MC: 当前 65% → 目标 75%+
- 话题覆盖率: 100%（种子 + AAM + LLM 兜底）

## 核心文件

- activation-field.ts: 9+信号激活场 + 话题分池路由
- aam.ts: AAM PMI 共现学习 + 动态话题扩展
- distill.ts: L1→L2→L3 蒸馏 + TopicNode
- benchmark-locomo.ts: LOCOMO benchmark + CNAS pipeline
- seeds/category_seeds_v2.json: C1/C2/C3 三层种子库（5030 个通用关键词）
- types.ts: Memory 接口（primary_c1/c2/c3 单标签字段）
